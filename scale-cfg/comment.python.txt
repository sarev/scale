This is the Python doc-comment format: a PEP 257 docstring. For Python it supersedes the generic
"Function doc-comments" shape above. Match it exactly.

Example, a function `cohort_verdict()`:

    """Judge a failed job's cohort, returning the verdict and the axes it was read across.

    The judgement is worst-first: an intermittent split (identical setup, opposite outcome) is reported
    before a clean suspect split, so the most actionable signal wins. It is pure: it computes over the rows
    it is given and adds no opinion the data does not support.

    Parameters:
    - rows: The comparable jobs, newest first.
    - sku: The product the cohort is defined for.

    Returns:
    - A `(verdict, axes)` pair; `verdict` is one of `intermittent`, `suspect` or `inconclusive`.

    Raises:
    - ValueError: When `rows` is empty.
    """

A trivial helper needs only the summary line:

    """Return the held credential, or an empty string when none is set."""

**Rules:**

1. Output exactly one docstring per turn, and nothing else: no code, no fences, no commentary. When asked for
   the body only, omit the `"""` delimiters.
2. The first line is a one-line imperative summary ending with a full stop. Then a blank line, then any
   elaboration (the reasoning, contracts, ordering constraints, edge cases), then the sections.
3. Document the parameters, the return value and the raised exceptions as bulleted lists under `Parameters:`,
   `Returns:` and `Raises:` headings, in that order. Omit any heading that would say nothing: no `Parameters:`
   for a method taking only `self`, no `Returns:` for a function returning `None`, no `Raises:` when nothing
   is raised.
4. Never restate the type. The signature already carries it; the prose carries the meaning, units and edge
   cases. Write `- count: How many checks to run.`, never `- count (int): ...`.
5. The prose must give insight, not paraphrase the signature. A docstring that only restates the signature in
   words is not worth writing; cut it to the summary line.
6. Plain prose with single backticks for inline code and identifiers (`None`, `path`, `/api/state`). No other
   Markdown: no bold, no headings, and no ad-hoc sections beyond the three named above.
7. Self-contained: cite nothing outside the source. No decision or issue identifiers, no document, plan or
   spec names, no commit hashes, and no change history ("was formerly", "now does", dates). State the reason
   itself. A reference to other source (a sibling method, a class) is fine.
8. British English spellings (`colour`, `behaviour`, `serialise`).
9. Wrap at about 120 columns.
10. Reformat an existing docstring that does not already follow this format.
